# Phase 07 — Reward Redemption

## Objective

Implement the reward catalog and redemption flow with robust double-spend prevention. The grading criteria specifically call out preventing "double spending" on concurrent clicks.

---

## Business Rules

1. Users can browse a catalog of rewards with point costs
2. Users can redeem a reward if they have enough points (received balance)
3. **Double-spending is not allowed** — rapid clicks or concurrent requests must not deduct twice
4. Rewards with limited stock must decrement correctly
5. Users can view their redemption history

---

## Backend Implementation

### API Endpoints

**GET /api/rewards** — List active rewards
- Returns: id, name, description, pointCost, imageUrl, stock, isActive
- Only active rewards shown

**POST /api/rewards** — Create reward (admin only)
- Input: name, description, pointCost, imageUrl, stock (-1 for unlimited)

**PATCH /api/rewards/:id** — Update reward (admin only)

**POST /api/redemptions** — Redeem a reward
- Input: rewardId, idempotencyKey (UUID, generated by client)
- Returns: redemption record

**GET /api/redemptions** — User's redemption history
- Paginated, ordered by created_at DESC

**GET /api/users/me/balance** — Current spendable point balance
- Derived from ledger: SUM(amount) WHERE user_id = current user

### Transaction Flow (Redeem Reward)

```
BEGIN TRANSACTION
  1. Check idempotency_key — if exists, return existing redemption (200, not 201)
  2. Lock user's balance (advisory lock or row lock on user)
  3. Calculate balance from ledger: SUM(amount) WHERE user_id = ?
  4. Verify balance >= reward's point_cost
  5. If reward has stock > 0: lock reward row, verify stock, decrement
  6. Create redemption record (status: completed)
  7. Create ledger entry: -point_cost (type: redemption)
COMMIT
```

### Idempotency Key Pattern

This is the key defense against double-spending:

- **Client side**: Generate a UUID v4 before showing the confirm dialog. Attach it to the request.
- **Server side**: UNIQUE constraint on `idempotency_key` column. If the key already exists, return the existing redemption instead of creating a new one.
- **On network retry**: Same key → same result. No double deduction.
- **On double-click**: Same key → first request creates, second returns existing.

Why this works:
- Even if two requests arrive simultaneously, the UNIQUE constraint ensures only one INSERT succeeds
- The losing request gets a constraint violation, catches it, and returns the existing row
- This is the same pattern used by Stripe, PayPal, and other payment systems

### Balance Calculation

- **Source of truth**: `point_ledger` table
- **Balance**: `SELECT SUM(amount) FROM point_ledger WHERE user_id = ?`
- **Optimization**: Cache balance in Redis, invalidate on any ledger write
- **Fallback**: If cache miss, calculate from ledger

## Frontend Implementation

### Rewards Catalog Page

- Grid layout of reward cards (responsive: 1→2→3→4 columns)
- Each card: image, name, description, point cost, stock indicator
- "Redeem" button on each card
- User's current balance displayed prominently at top
- Disabled state for rewards the user can't afford (point cost > balance)
- "Out of stock" badge for depleted rewards

### Redemption Flow

1. User clicks "Redeem" on a reward card
2. Confirmation dialog appears: "Redeem [Item Name] for [X] points?"
3. Client generates idempotency key (UUID)
4. On confirm → button shows loading spinner, becomes disabled
5. Send POST /api/redemptions with rewardId + idempotencyKey
6. Success → toast notification + balance updates + close dialog
7. Error → show error message in dialog (insufficient points, out of stock)

### Redemption History

- List view of past redemptions
- Each row: reward name, points spent, date, status
- Accessible from profile or rewards page

## Tests

**Unit Tests**:
- Balance calculation from ledger entries
- Idempotency check logic
- Stock decrement logic

**Integration Tests**:
- Full redemption flow: API → DB → ledger entry created → balance decreased
- Redeem with insufficient balance → 400
- Redeem out-of-stock item → 400
- Redeem inactive reward → 404

**Concurrency Tests (Critical)**:
- Same idempotency key sent twice → only one redemption created
- User has 500 points, fires 3 concurrent redemptions for 200pts each → max 2 succeed
- Reward has stock=1, two users redeem simultaneously → only one succeeds
